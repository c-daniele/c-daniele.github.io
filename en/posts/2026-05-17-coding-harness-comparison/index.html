Intro

A few weeks ago I kept reading the same two complaints, posted by different people on different platforms, almost word for word: Claude Code “burns tokens like there is no tomorrow”, and there is this new lightweight tool called pi that “just works” with a fraction of the context. Both claims are usually made without numbers.

So I did the most boring thing I could think of. I created a one-line dummy project, pointed three different coding agents at it, typed hi, and then captured every byte that left for the model. Same model in all three runs (Claude Haiku 4.5 on Bedrock), same gateway (LiteLLM with full request/response logging), same prompt.

The thesis I want to defend in this article is the following: the real differences between coding agents do not live in their UI or in their model choice — they live in the harness. The harness is the system prompt, the tool catalogue, the rendering of the project context, the cache strategy. It is what you actually pay for on every single turn. And once you look at it that way, the “pi is so much cheaper” / “Claude Code is so wasteful” narrative becomes, in my opinion, much more nuanced than what circulates in the community.

Why this comparison matters

Coding agents are getting compared on benchmarks all the time: SWE-bench, HumanEval, custom internal evals. Those numbers tell you something about the model behind the agent. They tell you very little about the harness, which is the part you can actually change.

The harness decides:

  • how much context you ship to the model on every single turn, before the user has even spoken;
  • how the model is taught to use tools (a 200-character bash description vs. a 10,000-character one is a different agent in practice);
  • whether prompt caching kicks in, and on which boundary;
  • which tools are even available out of the box, and how strongly they are guard-railed.

If you run agents at any non-trivial scale — multiple developers, long sessions, several iterations per task — the harness is where the cost lives. So I wanted real numbers, not vibes.

The setup

The project is intentionally minimal. One file, AGENTS.md:

This is an example project to make some experiment with coding agents. Key tech stack:
 - python (uv)
 - shell (zsh)
 - aws cli

Three agents, all pinned to the same model so the only moving part is the harness:

The headline numbers

Let’s start with the part everybody cares about: tokens and money for a single turn that does literally nothing useful.

A few things jump out:

  • Claude Code sends ~11x more prompt tokens than pi. That is a real, measurable difference, not a rumor.
  • OpenCode sits roughly in the middle, at about 4.8x pi and 0.44x Claude Code.
  • The first call is always more expensive, because cache creation is billed at a premium (around 1.25x the input rate for Anthropic models). The interesting question is what happens on the second turn — and that is where caching changes the picture entirely. We will get back to it.

So pi wins the headline. But the headline is not the article.

Where the tokens actually go

Two things stand out, and they are not what I expected.

The tool catalogue dominates. In every single agent, the tool descriptions and their JSON schemas weigh more than the base system prompt. For Claude Code, the tools alone are ~19k tokens — bigger than OpenCode’s entire request. The Bash tool description in Claude Code is 10,637 characters, roughly 2,600 tokens just to teach the model how to run shell commands.

The number of tools matters less than their average size. Claude Code ships 27 tools, OpenCode 12, pi 5. Cutting tool count by a factor of 5 (Claude Code → pi) cuts tool-catalogue cost by a factor of ~17. That is the verbosity tax.

OpenCode’s average tool is actually slightly bigger than Claude Code’s, because OpenCode is more aggressive in collapsing functionality into fewer tools. That single bash tool in OpenCode is almost the same size as Claude Code’s Bash, and todowrite is nearly 9k characters by itself. Claude Code looks heavier because it has more entries, not because each entry is more bloated.

In other words, “fewer tools” and “smaller harness” are not the same thing.

Why the big tools are big

Looking at the actual content of Bash is the most instructive part of this experiment. Both Claude Code and OpenCode treat the bash tool as a place to embed an entire operations manual: when not to use it (prefer Read/Edit/Grep over cat/sed/grep), how to quote paths with spaces, how to handle git operations, how to write a PR description, even a HEREDOC template for gh pr create.

That is not just verbosity. Those guard-rails exist because, on a real project, an unconstrained bash is the single most dangerous tool an agent has. The descriptions are the result of a long tail of incidents — agents cd-ing out of the workspace, agents running destructive rm -rf, agents inventing flags, agents committing on the wrong branch.

pi, by contrast, defines bash like this:

Execute a bash command in the current working directory. Returns stdout
and stderr. Output is truncated to last 2000 lines or 50KB (whichever
is hit first). If truncated, full output is saved to a temp file.
Optionally provide a timeout in seconds.

That is it. 248 characters. No prohibitions, no examples, no defensive playbook.

This gives pi a real token advantage on the first turn — and a real behavioural deficit on every turn after that. The model has no instruction to prefer dedicated tools over bash, no quoting guidance, no anti-footgun for git. It will happily run cat large_file.py instead of read, or grep -r instead of a glob tool, every time. The cheap system prompt becomes an expensive runtime, paid in tool calls and tokens you cannot see in this single-turn benchmark.

To be honest, this is the part of pi I am most skeptical about. Anyone using it on a serious project will end up reinventing those guard-rails — and at that point, you have rebuilt OpenCode in your AGENTS.md.

The skill manifest: Claude Code’s hidden bill

There is a smaller, but interesting, contributor in Claude Code that does not exist in the other two: the skill manifest.

Claude Code injects, as part of the user turn, a list of all installed skills with their full trigger descriptions. In my run, with default skills only, that block was about 3,000 characters (~750 tokens). Skills like claude-api, update-config, keybindings-help, simplify, loop, schedule, init, review, security-review are all advertised on every single call, even when the user just typed “hi”.

This is a deliberate design choice — it lets the model decide whether to invoke the Skill tool to load a heavier playbook. It is also a design choice that scales linearly with how many skills you install. If you are the kind of user that installs a dozen extra skills, you are paying for that catalogue on every turn.

Neither OpenCode nor pi has an equivalent in their default request. OpenCode’s skill tool is mentioned, but the skill list is not pre-injected. pi has no skill mechanism at all by default.

Caching, or: why turn-2 looks very different

The headline cost above is a worst case: cold start, first turn. The real cost of an agent over a session depends on how much of that big prompt gets cached and replayed.

Anthropic prompt caching has a simple economic shape: cache creation costs ~1.25x normal input, but cache reads cost ~0.1x. So as long as your harness reuses the same prefix on subsequent turns, the per-turn cost collapses by an order of magnitude.

Look at the cache fields in the metadata for our three runs:

Agentcache_control set?cache_creation_input_tokenscache_read_input_tokens
Claude Codeyes (system + last user block)28,3980
OpenCodenot in payload12,3660
piyes (system + user block)00

Claude Code is doing what the documentation tells you to do: it explicitly marks the system blocks and the latest user block with {"type": "ephemeral"}. The cache is created on this first call (28,398 tokens written), and on every subsequent call within the 5-minute window the model will pay 0.1x for the prefix instead of 1x.

OpenCode is more interesting. Its payload contains no cache_control at all — it uses the OpenAI-compatible style with plain string system and user content. And yet the response shows 12,366 cache-creation tokens. The most plausible explanation is that LiteLLM (or the underlying Bedrock provider) auto-injects cache breakpoints when it can. So OpenCode gets caching essentially for free, without doing anything explicit. That is a nice property if you trust your gateway, less nice if you ever change gateway.

pi is the surprise. It does set cache_control: ephemeral on its system block and on the user message. And yet cache_creation_input_tokens is 0. The cache simply did not engage. I have not fully diagnosed why — possible suspects include the way pi sets its blocks (only one breakpoint pair, but Anthropic supports up to four), some edge in the LiteLLM/Bedrock path, or a model-specific minimum that was not met. What I can say is that, on this exact request through this exact gateway, pi is paying 1x on every turn for content that Claude Code and OpenCode are paying 0.1x for after the first turn.

In other words, the headline ratio reverses very fast.

pi’s lead survives, but it shrinks from 11x to roughly 1.07x against OpenCode. And every turn that adds tool output to the conversation closes the gap further, because that delta is paid at 1x in all three agents. If pi’s caching ever does engage (and it should, in principle), the ranking flips.

The point is not that any specific number here is precise — there are simplifications all over this estimate. The point is the shape of the curve: with caching, the harness size matters dramatically less than it does on turn 1.

What I take away from this experiment

1. The harness is the product

When you compare coding agents, you are mostly comparing system prompts and tool catalogues. The model is the same. The architecture decisions — which tools to expose, how strongly to guard-rail them, what context to inject into the user turn, whether to flag cache breakpoints — are the agent.

2. “Lighter” is not free

pi looks dramatically cheaper on the first turn, and it is. But part of that saving is a deliberate decision not to teach the model how to behave. Anyone running pi on a serious codebase will end up adding rules through AGENTS.md and project conventions, and the gap will close. In the limit, you re-derive OpenCode.

I want to be clear: pi is a perfectly fine tool for what it is — a minimal, hackable harness for people who want to build their own. I would just not read its on-paper token count as a generic “this agent is more efficient”.

3. OpenCode is, in my opinion, the best default trade-off

It carries roughly half the prompt of Claude Code, has a comparable set of useful tools, and gets caching even without explicit cache_control in its payload (at least via LiteLLM). For most projects, that is the practical sweet spot.

4. Caching is not a feature you can take for granted

Two of three agents had caching engaged on this run. One did not, despite asking for it. If your cost model assumes 0.1x reads after turn 1, you should measure — every gateway, every provider, every harness version. Otherwise you are budgeting on a hope.

5. Tool descriptions are documentation for the model — and they are also a tax

The big bash and todowrite descriptions in Claude Code and OpenCode look excessive when you isolate them. They are also the accumulated lessons from a lot of bad agent runs. Trimming them is possible, but it is a deliberate engineering decision with a real risk profile, not a free lunch.

Limits of this experiment

  • One turn, one model, one prompt. A real session will have many turns, tool outputs, and follow-ups. The headline ratios will move.
  • One gateway. LiteLLM may be doing things (auto cache breakpoints, header rewriting) that I cannot fully see from the captured payloads. Running directly against Bedrock or Anthropic could give different numbers, especially for OpenCode and pi.
  • Default configurations only. All three agents can be configured to be lighter or heavier. The numbers above are the out-of-the-box experience, not a tuned one.
  • No quality measurement. I deliberately did not test whether the agents do the job better. That is a different article, with very different methodology.

If you want to argue with any of this, the artifacts are public. The interesting part is not the absolute numbers, it is the breakdown — and what it tells you about where to spend your engineering effort if you are designing your own coding agent in 2026.
