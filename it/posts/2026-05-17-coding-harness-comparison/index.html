Intro

Nelle ultime settimane mi sono ritrovato a leggere le stesse due lamentele, postate da persone diverse su piattaforme diverse, quasi parola per parola: Claude Code “brucia token come se non ci fosse un domani”, e c’è questo nuovo tool leggero chiamato pi che “funziona alla grande” con una frazione del contesto. Entrambe le affermazioni sono di solito fatte senza numeri.

Ho fatto allora la cosa più noiosa che mi è venuta in mente. Ho creato un progetto dummy da una riga, ho puntato tre coding agent diversi su quello, ho scritto hi e ho catturato ogni byte uscente verso il modello. Stesso modello in tutti e tre i run (Claude Haiku 4.5 su Bedrock), stesso gateway (LiteLLM con logging completo di richieste e risposte), stesso prompt.

La tesi che voglio difendere in questo articolo è la seguente: le vere differenze tra coding agent non vivono nella loro UI o nella scelta del modello — vivono nell’harness. L’harness è il system prompt, il catalogo dei tool, il modo in cui viene reso il contesto del progetto, la strategia di caching. È quello che paghi davvero su ogni singolo turno. E una volta che la guardi così, la narrazione “pi è molto più economico” / “Claude Code è uno spreco” diventa, a mio avviso, molto più sfumata di quello che circola nella community.

Perché questo confronto conta

I coding agent vengono confrontati continuamente sui benchmark: SWE-bench, HumanEval, eval interni custom. Quei numeri ti dicono qualcosa sul modello che sta dietro all’agente. Ti dicono molto poco sull’harness, che invece è la parte che puoi davvero modificare.

L’harness decide:

  • quanto contesto spedisci al modello su ogni singolo turno, prima ancora che l’utente abbia parlato;
  • come al modello viene insegnato a usare i tool (una descrizione di bash da 200 caratteri o una da 10.000 caratteri sono in pratica due agent diversi);
  • se il prompt caching si attiva, e su quale boundary;
  • quali tool sono addirittura disponibili out of the box, e con quanti guardrail.

Se fai girare gli agent a una scala non triviale — più sviluppatori, sessioni lunghe, diverse iterazioni per task — l’harness è dove vive il costo. Per questo volevo numeri veri, non sensazioni.

Il setup

Il progetto è volutamente minimale. Un solo file, AGENTS.md:

This is an example project to make some experiment with coding agents. Key tech stack:
 - python (uv)
 - shell (zsh)
 - aws cli

Tre agent, tutti agganciati allo stesso modello in modo che l’unica variabile sia l’harness:

AgentVersioneSorgente

Tutti e tre puntati allo stesso gateway LiteLLM, tutti e tre che chiamano eu.anthropic.claude-haiku-4-5-20251001-v1:0 su AWS Bedrock. Singolo messaggio utente: hi.

LiteLLM ha loggato il payload completo di ogni chiamata: corpo della richiesta, corpo della risposta, e un blocco di metadata con i conteggi di token, il dettaglio della cache e il costo. Gli artefatti catturati sono nella repo di accompagnamento per chi voglia riprodurre l’esperimento o discutere la mia lettura.

I numeri di copertina

Partiamo dalla parte che interessa a tutti: token e soldi per un singolo turno che letteralmente non fa niente di utile.

Saltano subito all’occhio alcune cose:

  • Claude Code spedisce ~11 volte più prompt token di pi. È una differenza reale e misurabile, non una voce di corridoio.
  • OpenCode si piazza grossomodo nel mezzo, a circa 4,8x pi e 0,44x Claude Code.
  • La prima chiamata è sempre la più costosa, perché la creazione della cache è fatturata a un premium (circa 1,25x il rate normale di input per i modelli Anthropic). La domanda interessante è cosa succede al secondo turno — ed è lì che il caching cambia completamente lo scenario. Ci torno tra poco.

Quindi pi vince la copertina. Ma la copertina non è l’articolo.

Dove finiscono davvero i token

Ho scomposto ogni richiesta nei suoi componenti reali — system prompt, catalogo dei tool (descrizione + JSON schema), e qualunque contesto extra iniettato come parte del turno utente. L’approssimazione qui è grezza (caratteri/4), ma segue i prompt_tokens misurati con un errore di pochi punti percentuali.

Due cose risaltano, e non sono quelle che mi aspettavo.

Il catalogo dei tool domina. In ogni singolo agent, le descrizioni dei tool e i loro JSON schema pesano più del system prompt di base. Per Claude Code, i soli tool sono ~19k token — più dell’intera richiesta di OpenCode. La descrizione del tool Bash in Claude Code è di 10.637 caratteri, circa 2.600 token solo per insegnare al modello come eseguire comandi shell.

Il numero di tool conta meno della loro dimensione media. Claude Code spedisce 27 tool, OpenCode 12, pi 5. Ridurre il numero di tool di un fattore 5 (Claude Code → pi) riduce il costo del catalogo di un fattore ~17. È la tassa sulla verbosità.

Il tool medio di OpenCode è in realtà leggermente più grande di quello di Claude Code, perché OpenCode è più aggressivo nel collassare funzionalità in pochi tool. Quel singolo tool bash di OpenCode è quasi della stessa dimensione del Bash di Claude Code, e todowrite da solo sfiora i 9k caratteri. Claude Code sembra più pesante perché ha più voci, non perché ogni voce sia più gonfia.

In altre parole, “meno tool” e “harness più piccolo” non sono la stessa cosa.

Perché i tool grandi sono grandi

Guardare il contenuto reale di Bash è la parte più istruttiva di questo esperimento. Sia Claude Code che OpenCode trattano il tool bash come il posto in cui infilare un intero manuale operativo: quando non usarlo (preferire Read/Edit/Grep rispetto a cat/sed/grep), come quotare path con spazi, come gestire operazioni git, come scrivere una descrizione di PR, perfino un template HEREDOC per gh pr create.

Non è solo verbosità. Quei guardrail esistono perché, in un progetto reale, un bash senza vincoli è il tool singolo più pericoloso che un agent abbia. Le descrizioni sono il risultato di una lunga coda di incidenti — agent che fanno cd fuori dal workspace, agent che eseguono rm -rf distruttivi, agent che si inventano flag, agent che committano sul branch sbagliato.

pi, al contrario, definisce bash così:

Execute a bash command in the current working directory. Returns stdout
and stderr. Output is truncated to last 2000 lines or 50KB (whichever
is hit first). If truncated, full output is saved to a temp file.
Optionally provide a timeout in seconds.

Tutto qui. 248 caratteri. Niente divieti, niente esempi, niente playbook difensivo.

Questo dà a pi un vero vantaggio sui token al primo turno — e un vero deficit comportamentale su ogni turno successivo. Il modello non ha alcuna istruzione per preferire i tool dedicati a bash, nessuna guida sul quoting, nessun anti-piedone per git. Eseguirà allegramente cat large_file.py invece di read, o grep -r invece di un tool di glob, ogni volta. Il system prompt economico diventa un runtime costoso, pagato in tool call e token che non vedi in questo benchmark single-turn.

A essere onesto, è la parte di pi su cui sono più scettico. Chiunque lo usi su un progetto serio finirà per reinventare quei guardrail — e a quel punto, hai ricostruito OpenCode dentro il tuo AGENTS.md.

Lo skill manifest: il conto nascosto di Claude Code

C’è un contributo più piccolo, ma interessante, presente in Claude Code e assente negli altri due: lo skill manifest.

Claude Code inietta, come parte del turno utente, una lista di tutte le skill installate con le loro descrizioni di trigger complete. Nel mio run, con le sole skill di default, quel blocco era di circa 3.000 caratteri (~750 token). Skill come claude-api, update-config, keybindings-help, simplify, loop, schedule, init, review, security-review vengono pubblicizzate a ogni singola chiamata, anche quando l’utente ha appena scritto “hi”.

È una scelta progettuale deliberata — permette al modello di decidere se invocare il tool Skill per caricare un playbook più pesante. È anche una scelta progettuale che scala linearmente con il numero di skill che installi. Se sei il tipo di utente che installa una dozzina di skill extra, stai pagando quel catalogo a ogni turno.

Né OpenCode né pi hanno un equivalente nella loro richiesta di default. Il tool skill di OpenCode è menzionato, ma la lista delle skill non viene pre-iniettata. pi non ha alcun meccanismo di skill di default.

Caching, ovvero: perché il turno-2 ha tutta un’altra faccia

Il costo di copertina visto sopra è un caso pessimo: cold start, primo turno. Il costo reale di un agent in una sessione dipende da quanto di quel grosso prompt viene messo in cache e replicato.

Il prompt caching di Anthropic ha una forma economica semplice: la creazione di cache costa ~1,25x dell’input normale, ma la lettura di cache costa ~0,1x. Quindi finché il tuo harness riusa lo stesso prefisso sui turni successivi, il costo per turno crolla di un ordine di grandezza.

Guarda i campi cache nei metadata dei nostri tre run:

Agentcache_control impostato?cache_creation_input_tokenscache_read_input_tokens
Claude Codesì (system + ultimo blocco user)28.3980
OpenCodenon nel payload12.3660
pisì (system + blocco user)00

Claude Code fa quello che la documentazione dice di fare: marca esplicitamente i blocchi di system e l’ultimo blocco utente con {"type": "ephemeral"}. La cache viene creata su questa prima chiamata (28.398 token scritti), e su ogni chiamata successiva entro la finestra di 5 minuti il modello pagherà 0,1x per il prefisso invece di 1x.

OpenCode è più interessante. Il suo payload non contiene alcun cache_control — usa lo stile OpenAI-compatible con system e user come stringhe semplici. E tuttavia la risposta mostra 12.366 token di cache creation. La spiegazione più plausibile è che LiteLLM (o il provider Bedrock sottostante) inietti automaticamente i breakpoint di cache quando può. Quindi OpenCode ottiene il caching essenzialmente gratis, senza fare nulla di esplicito. È una bella proprietà se ti fidi del tuo gateway, meno bella se mai cambierai gateway.

pi è la sorpresa. Imposta cache_control: ephemeral sul suo blocco system e sul messaggio utente. Eppure cache_creation_input_tokens è 0. La cache semplicemente non è entrata in azione. Non ho diagnosticato a fondo il perché — i sospetti possibili includono il modo in cui pi imposta i suoi blocchi (un solo coppia di breakpoint, mentre Anthropic ne supporta fino a quattro), qualche edge nel path LiteLLM/Bedrock, o un minimo specifico del modello che non è stato raggiunto. Quello che posso dire è che, su questa esatta richiesta attraverso questo esatto gateway, pi sta pagando 1x su ogni turno per contenuti che Claude Code e OpenCode pagano 0,1x dopo il primo turno.

In altre parole, il rapporto di copertina si ribalta molto in fretta.

Rifare i conti per una sessione di 10 turni

Faccio uno schizzo del costo su una sessione da 10 turni, assumendo che a ogni turno l’harness spedisca il suo prefisso base più un piccolo delta. Usando la dimensione del prompt misurata di ogni agent come prefisso messo in cache e ipotizzando letture di cache sui turni 2-10:

        T1CC["Claude Code: ~28k @ 1.25x"]
        T1OC["OpenCode: ~12k @ 1.25x"]
        T1PI["pi: ~2.5k @ 1x"]

    subgraph "Turni 2..10 (warm)"
        T2CC["Claude Code: ~28k @ 0.1x"]
        T2OC["OpenCode: ~12k @ 0.1x"]
        T2PI["pi: ~2.5k @ 1x (no cache)"]

Numeri di massima, mettendo da parte il costo di completion per chiarezza:

AgentCosto input turno 1Costo input turni 2-10 (cad.)Costo input 10 turni
Claude Code$0,039~$0,003~$0,066
OpenCode$0,017~$0,0014~$0,030
pi (no cache)$0,0028$0,0028~$0,028

Il vantaggio di pi sopravvive, ma si riduce da 11x a circa 1,07x rispetto a OpenCode. E ogni turno che aggiunge output di tool alla conversazione chiude ulteriormente il gap, perché quel delta viene pagato a 1x in tutti e tre gli agent. Se la cache di pi dovesse mai entrare in funzione (e in teoria dovrebbe), la classifica si ribalta.

Il punto non è che un numero specifico qui sia preciso — ci sono semplificazioni ovunque in questa stima. Il punto è la forma della curva: con il caching, la dimensione dell’harness conta drasticamente meno di quanto conti al turno 1.

Cosa porto a casa da questo esperimento

1. L’harness è il prodotto

Quando confronti coding agent, stai confrontando soprattutto system prompt e cataloghi di tool. Il modello è lo stesso. Le decisioni architetturali — quali tool esporre, quanti guardrail metterci, che contesto iniettare nel turno utente, se segnalare i breakpoint di cache — sono l’agent.

2. “Più leggero” non è gratis

pi sembra drasticamente più economico al primo turno, e lo è. Ma parte di quel risparmio è una decisione deliberata di non insegnare al modello come comportarsi. Chiunque lo faccia girare su una codebase seria finirà per aggiungere regole tramite AGENTS.md e convenzioni di progetto, e il gap si chiuderà. Al limite, ti riconverti su OpenCode.

Voglio essere chiaro: pi è un tool perfettamente valido per quello che è — un harness minimale e hackerabile per chi vuole costruirsi il proprio. Eviterei semplicemente di leggere il suo conteggio token sulla carta come una generica “questo agent è più efficiente”.

3. OpenCode è, secondo me, il miglior trade-off di default

Porta circa metà del prompt di Claude Code, ha un set comparabile di tool utili, e ottiene il caching anche senza cache_control esplicito nel suo payload (almeno via LiteLLM). Per la maggior parte dei progetti, è il punto di equilibrio pratico.

4. Il caching non è una feature da dare per scontata

Due agent su tre avevano il caching attivo in questo run. Uno no, nonostante lo richiedesse. Se il tuo modello di costo assume letture a 0,1x dopo il turno 1, dovresti misurare — ogni gateway, ogni provider, ogni versione di harness. Altrimenti stai facendo budget su una speranza.

5. Le descrizioni dei tool sono documentazione per il modello — e sono anche una tassa

Le grandi descrizioni di bash e todowrite in Claude Code e OpenCode sembrano eccessive quando le isoli. Sono anche le lezioni accumulate da molti run di agent andati male. Tagliarle è possibile, ma è una decisione di engineering deliberata con un profilo di rischio reale, non un pasto gratis.

Limiti di questo esperimento

  • Un turno, un modello, un prompt. Una sessione reale avrà molti turni, output di tool, follow-up. I rapporti di copertina si muoveranno.
  • Un solo gateway. LiteLLM potrebbe star facendo cose (auto cache breakpoint, riscrittura header) che non vedo del tutto dai payload catturati. Girando direttamente contro Bedrock o Anthropic i numeri potrebbero essere diversi, soprattutto per OpenCode e pi.
  • Solo configurazioni di default. Tutti e tre gli agent possono essere configurati per essere più leggeri o più pesanti. I numeri qui sopra sono l’esperienza out of the box, non quella tunata.
  • Nessuna misura di qualità. Volutamente non ho testato se gli agent fanno il lavoro meglio. È un altro articolo, con metodologia molto diversa.

Se vuoi discutere qualcosa di tutto questo, gli artefatti sono pubblici. La parte interessante non sono i numeri assoluti, è la scomposizione — e quello che ti dice su dove spendere lo sforzo di engineering, se nel 2026 stai progettando il tuo coding agent.
